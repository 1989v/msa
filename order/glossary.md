## 1. Bounded Context Overview

Order BC는 **주문 생성·상태 전이·외부 결제 연동**을 담당하는 커머스 핵심. 주문 시 Product BC에 상품 유효성·재고 차감 API를 호출하며, 결제는 외부 PaymentPort로 위임하고 CircuitBreaker로 장애를 격리한다 (ADR-0015). 주문 완료·취소는 Kafka 토픽 `order.order.completed` / `order.order.cancelled`로 발행되어 후속 BC(inventory confirm, fulfillment, analytics)가 소비한다.

## 2. Aggregates & Entities

### Order
**Type.** Aggregate
**Definition.** 고객 결제 의사 표시로 생성된 거래 단위. 여러 OrderItem을 보유하며 OrderStatus 라이프사이클을 가짐.
**Lifecycle.** `create(userId, items)` → PENDING → {`complete()` → COMPLETED | `cancel()` → CANCELLED}
**Invariants.**
- `items.isNotEmpty()` — 빈 주문 금지
- `userId.isNotBlank()` — 익명 주문 금지
- `complete()` / `cancel()` 은 **PENDING 상태에서만** 가능 (불가역 종료 상태)
- `totalAmount` 는 OrderItem 의 subtotal 합으로 **자동 계산** (저장 필드 아님)
**Related events.** emits: `order.order.completed`, `order.order.cancelled`
**Avoid.** Purchase, Transaction, FulfillmentOrder(다른 BC), OrderCommand(quant BC)
**Code.** `com.kgd.order.domain.order.model.Order`
**Found in.** `order/domain/order/model/`

### OrderItem
**Type.** Entity (Order Aggregate 내부)
**Definition.** Order 내의 개별 상품 라인. 수량·단가·자동 계산 소계 보유.
**Lifecycle.** Order와 동일 (Order 생성 시 함께 생성, 별도 라이프사이클 없음)
**Invariants.**
- `quantity > 0` — 0개 이하 라인 금지
- `subtotal = unitPrice.amount * quantity` (자동 계산 속성, BigDecimal 곱)
**Avoid.** LineItem, ProductLine
**Code.** `com.kgd.order.domain.order.model.OrderItem`

## 3. Value Objects

### Money
**Type.** Value Object (`@JvmInline value class`)
**Definition.** 금액 표현. BigDecimal 래퍼.
**Invariants.** `amount >= BigDecimal.ZERO` (0 허용)
**Avoid.** Amount, Price
**Code.** `com.kgd.order.domain.order.model.Money`

> **Cross-context 차이**: product BC의 Money는 `amount > ZERO` (0 금지). 본 BC는 `>= ZERO` (0 허용 — 무료 상품/쿠폰 100% 차감 등 시나리오?). Flagged #1.

## 5. Domain Events

### order.order.completed / order.order.cancelled (Kafka)
**Type.** Domain Event
**Definition.** Order 상태 전이를 외부에 알림.
**Code.** `com.kgd.order.app.event.OrderEvents`
**Trigger.** `Order.complete()` / `Order.cancel()` 직후 → OrderEventAdapter → Kafka
**Consumers.**
- inventory BC: `confirmed` → ConfirmStock, `cancelled` → ReleaseStock
- fulfillment BC: `completed` → CreateFulfillmentUseCase
- analytics BC: 두 이벤트 모두 (지표 산출)

## 6. Policies & Invariants

| 출처 | 규칙 |
|---|---|

| `OrderNotFoundException` | id 조회 실패 |
| `InvalidOrderStatusException` | 상태 검증 실패 (PENDING 외에서 전이 시도) |

## 7. Ports

### Outbound

#### OrderRepositoryPort / OrderRepositoryAdapter
JPA 영속화.

#### PaymentPort / PaymentAdapter
**Definition.** 외부 결제 시스템 호출 추상화. CircuitBreaker 적용 (ADR-0015).

#### ProductPort / ProductAdapter
**Definition.** Product BC 동기 호출 (상품 유효성 + 재고 차감 트리거). WebClient 기반.

#### OrderEventPort / OrderEventAdapter
**Definition.** OrderEvents Kafka 발행.

## 11. Cross-Context Integration

### Emitted Events
- `order.order.completed` → inventory(ConfirmStock), fulfillment(CreateFulfillment), analytics
- `order.order.cancelled` → inventory(ReleaseStock), analytics

### Consumed Events / API Calls
- → product BC `ProductPort` (sync call, 상품 유효성·재고 차감)
- → 외부 결제 시스템 `PaymentPort` (CircuitBreaker)
- ← `OrderEventConsumer` (외부 이벤트 수신, 용도 TBD)

### Shared Terms
| 용어 | 본 BC | 다른 BC | 해결 |
|---|---|---|---|
| **Order** | 고객 주문 (commerce) | quant BC: 거래소 주문 (OrderCommand), fulfillment BC: FulfillmentOrder | 본 BC만 단순히 "Order" |
| **Money** | `>= 0` 허용 | product BC: `> 0` | **🔴 정의 불일치** (Flagged #1) |
| **OrderStatus** | PENDING/COMPLETED/CANCELLED | quant BC도 `OrderStatus` 보유 (의미 다름) | 패키지로 구분 |
| **OrderItem** | 본 BC 전용 | — | — |
