## 1. 영상 핵심 주제 / 한 줄 요약

> "원본 데이터(RDB) + 워크로드별 보조 저장소(Elasticsearch / Redis / Vector DB / BigQuery 등)" 라는 하이브리드 DB 패턴을 일관된 메타 시각으로 설명하고, 검색 엔진이 그 패턴의 가장 대표적인 적용 사례임을 강조한다.

표면적으로는 "왜 검색은 검색엔진(Elasticsearch)에 맡겨야 하는가?" 라는 질문에서 출발하지만, 본질은 **Polyglot Persistence 의 실무적 적용 원칙**과 **Source-of-Truth + 파생 저장소 패턴**의 정의/조회 흐름/리스크/도입 단계를 정리하는 데 있다.

## 2. 섹션별 내용 정리 (광고 구간 없음 — 5번 항목 참조)

### 2-1. (00:00 ~ 01:07) 도입 — 왜 RDB 외에 Redis/ES 를 같이 쓰나
- 화자의 실무 경험: 게임 회사에서 자체 커뮤니티(수십만 동접 예측) 설계.
- 검색 대상 범위: 6개월 / 12개월 / 전체 누적 게시물 → 시간이 갈수록 무한 증가.
- 요구된 검색 기능:
  - 다중 필드 검색 (제목/작성자/내용/태그)
  - 본문 내용(전문) 검색
  - 한국어 형태소 분석
  - 오타 허용(typo tolerance)
  - 자동완성

### 2-2. (01:07 ~ 02:13) 의사결정 — RDB Full-Text vs Elasticsearch
- 비교 후 **Elasticsearch** 채택. 단, **원본 데이터는 RDB 유지**.
- RDB 를 SoR(Source of Record)로 둔 이유:
  - 트랜잭션 / 정합성 보장
  - 조직의 데이터 거버넌스(OLAP, BI)가 RDB 기반으로 정렬되어 있음
- ES 를 도입한 이유:
  - 강력한 검색 기능
  - 이벤트/정기 점검 시 트래픽 폭주 대비
  - **RDB 검색 부하 분리**

### 2-3. (02:13 ~ 03:21) RDB 가 부족한 대표 워크로드 3가지
| 워크로드 | RDB 한계 | 보조 저장소 |
|---|---|---|
| 게시판 본문 검색 | `LIKE '%키워드%'` → 인덱스 사용 불가, Full Table Scan | Elasticsearch |
| 인기글 좋아요 (쓰기 폭주) | row 단위 락 경합 → 트랜잭션 지연/전파 | Redis (싱글 스레드, write 빠름) |
| 실시간 랭킹 (정렬+조회) | 정렬 비용, 특정 순위 페치 비용 | Redis Sorted Set |
| 위치 기반 검색 ("내 주변 1km") | 일반 인덱스로는 비효율 | PostGIS / Redis GEO |

### 2-4. (03:21 ~ 04:28) "원본 데이터 + 보조 저장소" 패턴 정의
- **Source of Truth (원본)**: 정합성 측면에서 가장 신뢰할 수 있는 기준 데이터.
- **보조 저장소(파생)**: 특정 워크로드에 최적화된 저장소.
- 원본은 꼭 RDB 일 필요 없음 — 문서/이미지/객체 저장소도 가능.
- 보조 저장소 예시: Elasticsearch, Redis, Vector DB(의미 검색), BigQuery(분석).
- **핵심 원칙 (반복 강조)**: 보조 저장소의 데이터가 사라져도 **원본에서 재구성 가능**해야 한다.
- 이 패턴은 상위 개념인 **Polyglot Persistence** 의 한 실현 형태.

### 2-5. (04:28 ~ 05:35) 2단계 조회 패턴 (Two-Phase Lookup)
검색 흐름:
1. ES 에서 후보 게시물 ID 목록 조회 (분석기 + 역색인으로 빠른 매칭)
2. 그 ID 로 RDB 에 인덱스 쿼리 → 실제 본문/메타데이터 fetch
3. 애플리케이션에서 조립 후 반환

→ "**먼저 찾고(narrow), 다음으로 가져온다(fetch)**" 구조.

### 2-6. (05:35 ~ 06:42) 다중 기준 랭킹 사례
- Redis Sorted Set 에 **1차 기준 점수**로 후보 정렬/축소.
- 후보 ID 로 RDB 또는 Redis Hash 에서 가중치/시간/카테고리 등 부가정보 조회.
- 애플리케이션에서 **재정렬(re-rank)** 후 응답.
- 검색·랭킹 모두 동일한 메타 패턴: 후보를 보조 저장소에서 추려 원본/보강 저장소에서 채워온다.

### 2-7. (06:42 ~ 08:19) 실무 사례 7가지 메타 정리
| # | 워크로드 | 보조 저장소 | 비고 |
|---|---|---|---|
| 1 | 본문/상품/로그 검색 | Elasticsearch | 게시판, 이커머스, 로그 |
| 2 | 랭킹(리더보드, 다중기준) | Redis Sorted Set | 게임 리더보드 등 |
| 3 | 캐시 (Look-aside) | Redis | 세션, 프로필, 권한, 코드 테이블 |
| 4 | 카운터/좋아요 | Redis | 락 경합 회피, 쓰기 최적화 |
| 5 | 위치 기반 | Redis GEO / PostGIS | 주변 검색 |
| 6 | 의미(Semantic) 검색 | Pinecone / pgvector | Vector DB |
| 7 | 데이터 분석 | BigQuery / ClickHouse | 운영 트래픽과 분리 |

→ "도구는 바뀌어도 패턴 원리는 동일." **메타 시각**(개별 사례 → 관통하는 공통 구조)을 가지라는 것이 핵심 메시지.

### 2-8. (08:19 ~ 09:23) "무조건 분리"의 함정 — Just Use Postgres
- PostgreSQL 단독으로도 가능한 영역이 매우 넓음:
  - 벡터 검색 (pgvector)
  - 위치 기반 (PostGIS)
  - Full-Text Search
  - JSON 문서
  - LISTEN/NOTIFY 알림
  - 시계열 확장 (TimescaleDB)
- **"Just Use Postgres"**: 작은 팀/초기 트래픽에서는 단일 DB 가 더 빠른 가치 창출.
- 보조 저장소 추가 시 비용:
  - 운영 비용 ↑
  - **Dual Write 정합성 문제**
  - 학습 비용 (팀 전체)
  - 인프라/라이선스 비용
  - 장애 시 디버깅 복잡도 ↑

### 2-9. (09:23 ~ 10:29) 점진적 도입 로드맵
| 단계 | 트래픽/패턴 | 권장 |
|---|---|---|
| MVP / 초기 | 작고 한정적 | 단일 DB + 확장 기능(Postgres) |
| 성장기 | 특정 조회/트랜잭션 부하 누적 | **Redis** 먼저 도입 |
| 확장기 | 검색/분석 워크로드 부각 | Elasticsearch + 분석 도구 분리 |
| 대규모 | 높은 트래픽, 다양한 도메인 | 본격 Polyglot, **Outbox / CDC** 로 정합성 |

핵심 원칙: **"병목이 증명되기 전에는 분리하지 말자."**

### 2-10. (10:29 ~ 11:35) 정합성 — Eventual Consistency 와 Dual Write
- Dual Write 위험: RDB 기록 성공 + ES 기록 실패 → 두 저장소 데이터 어긋남.
- 보조 저장소 구조의 권장 일관성 모델: **Eventual Consistency (최종적 일관성)**.
- 강한 일관성(RDB) ↔ 최종적 일관성(보조 저장소) 대비 — 비유: YouTube 조회수.
- 동기화 방향: **원본 → 보조** 단방향이 안전.
- 보조 저장소는 언제든 **원본에서 재구성 가능**해야 한다(반복 강조).

### 2-11. (11:35 ~ 12:59) 핵심 정리 (마무리)
- 원본+보조 저장소 패턴 = **RDB 의 장점 유지 + 특화 영역 보완**.
- 모든 사례(검색/랭킹/캐시/카운터/위치/벡터/분석) 가 동일한 패턴의 변주.
- 분리는 공짜가 아니다 → 단일 DB 시작 후 **병목 증명 시 점진적으로** 확장.
- Dual Write 정합성은 **반드시 별도 설계** (Outbox / CDC).
- 마무리 명언: **"올바른 도구를 올바른 자리에. 아키텍처 역량은 도구를 많이 아는 것이 아니라 적절한 도구를 선택하는 안목에서 나온다."**

## 3. 검색엔진 관련 핵심 개념/기술 요약

### 3-1. 왜 RDB 의 LIKE 검색이 한계인가
- `LIKE '%foo%'` 는 인덱스 사용 불가(접두 와일드카드). Full Table Scan 발생.
- B-Tree 인덱스는 prefix 매칭 위주이며 "단어/형태소" 단위 매칭이 아니라 문자열 매칭.

### 3-2. Elasticsearch (그리고 OpenSearch) 가 메우는 영역
- **분석기(Analyzer) + 역색인(Inverted Index)**: 토큰 단위 매칭, 빠른 검색.
- **한국어 형태소 분석**: nori (ES) / Seunjeon (구) / nori (OpenSearch 도 동일).
- **오타 허용**: Levenshtein 기반 fuzzy query.
- **자동완성**: completion suggester / search-as-you-type.
- **다중 필드 검색**: `multi_match`.
- **부하 격리**: 검색 전용 클러스터 = RDB 의 OLTP 부하와 분리.

### 3-3. 2단계 조회 패턴 (Two-Phase Lookup) in 검색
```
[Client] → [API] → [ES: ID 후보 추출] → [RDB: PK 인덱스 조회 / 본문 fetch] → [응답 조립]
```
- ES 는 **선택(filter/rank) 책임**, RDB 는 **정확한 데이터(상태/관계) 책임**.
- ES 에 모든 필드를 다 넣지 않아도 됨 → 인덱스 크기/리프레시 비용 ↓.

### 3-4. 인덱싱 정합성 (영상에서 함의된 부분)
- 게시물 등록 시 RDB + ES 양쪽 기록 → Dual Write 위험.
- 안전한 방법: **Outbox + Kafka + ES Sink** 또는 **Debezium CDC → Kafka → ES**.
- 동기화 단방향: SoR(RDB) → ES(파생).
- ES 데이터 손실 시 → RDB 로부터 **재인덱싱(reindex)** 가능해야 함.

### 3-5. 언제 ES 가 아니라 다른 도구가 답인가
- 의미(임베딩) 기반 검색 → Vector DB (pgvector / Pinecone) 또는 ES 8 의 dense_vector.
- 위치 기반 → PostGIS / Redis GEO.
- 단순 캐시/카운터/랭킹 → Redis.
- 단순 키워드 + 소규모 → Postgres FTS 로도 충분.

## 4. 백엔드 시니어 관점 시사점 (이 프로젝트 search 서비스 / ES / OpenSearch 컨텍스트)

이 영상의 메시지를 그대로 우리 commerce platform 에 매핑하면 다음 항목이 점검 포인트가 된다.

### 4-1. SoR / 파생 저장소 경계 점검
- `product` 서비스: MySQL 이 SoR, Kafka 발행 → `search` 서비스가 ES 색인. **이미 영상 권고와 정합**.
- 점검: search 서비스가 자체 보유 데이터(예: 색인 메타, 가중치 튜닝)가 SoR 처럼 굳어지지 않도록 — "재구성 가능성"이 살아 있어야 한다.
- ADR-0029 / `docs/conventions/idempotent-consumer.md` 의 Idempotent Consumer 규약은 ES Sink 의 재처리 안전성과 직결된다.

### 4-2. Dual Write 금지 원칙
- 영상의 "RDB 와 ES 에 동시에 쓰기" → 우리 컨벤션 (`docs/conventions/transactional-usage.md`) 의 "외부 IO 분리" 와 일치.
- 권고: product 도메인 트랜잭션 안에서 ES 호출 금지 → **Outbox / Kafka 발행** 으로 비동기 위임. 이미 우리 아키텍처는 이 방향. 신규 서비스 추가 시 동일 규약 강제.

### 4-3. 2단계 조회의 의식적 적용
- search 서비스에서 ES 가 응답하는 것은 **ID + 매칭 점수 + 하이라이트** 정도로 제한.
- 상품 상세/재고/가격 같은 **변동성 큰 + 정합성 민감한** 필드는 product/inventory API 로 fetch (또는 Redis 캐시 hit). ES 에 가격을 박아 넣으면 가격 변경 시 부정합 + 재색인 폭증.
- → 영상의 "**ES 는 후보를 추려라, 본문은 RDB 에서**" 원칙을 코드 리뷰 체크리스트로 명문화 가능.

### 4-4. Eventual Consistency 의 SLA 설계
- ADR-0025 latency budget 과 결합: "**색인 lag 의 P99 SLA**" 를 명시해야 한다.
  - 예: 상품 변경 → ES 반영까지 P99 < 5s.
- 사용자 경험 관점: 결제 직후 검색 결과에 안 보이는 케이스의 UX 처리(개인화 fallback 등) 를 product/search 협업으로 정리.

### 4-5. ES vs OpenSearch 도구 선택
- 우리 인프라에 ES + OpenSearch 가 모두 있는 상황 → 영상의 "**도구를 많이 아는 것이 아니라 선택하는 안목**" 메시지가 그대로 적용.
- 점검: 두 엔진을 모두 운영하는 비용(학습, 모니터링, 백업, 보안 패치) 이 정당화되는가? 기능적으로 통합 가능한가? → ADR 후보로 검토 가치 있음 ("`docs/adr/`: ES vs OpenSearch 일원화 검토").

### 4-6. "Just Use Postgres" 의 역적용
- 신규 서비스(quant, code-dictionary, gifticon 등) Phase 1 단계에서는 영상 권고대로 **단일 DB + 확장**으로 시작 → 병목 증명 후 분리.
- 현재 `quant` 가 Phase 1 진행 중이라면 이 가이드라인을 적용해 ES/Redis 도입을 "필요 증명" 시점까지 미루는 결정의 근거로 사용 가능.

### 4-7. 보조 저장소 재구성 가능성 = 운영 안전망
- 운영 절차서: **"ES 인덱스 전체 삭제 → 재색인" 시나리오를 복구 훈련(DR drill)** 항목으로 넣을 가치.
- 우리 백업 정책(`docker/backup/README.md`) 은 RDB binlog PITR 중심 → ES 는 SoR 가 아니므로 별도 백업이 필수가 아니라는 영상의 원칙과 정합. 단, **재색인 시간 = 복구 RTO** 임을 명시해야 한다.

### 4-8. Polyglot 도입의 점진성
- 영상의 4단계 로드맵을 우리 플랫폼의 신규 서비스 도입 가이드로 차용 가능:
  1. MVP: MySQL 단독
  2. 성장기: Redis 추가
  3. 확장기: ES/OpenSearch + ClickHouse(analytics) 분리
  4. 대규모: Outbox / CDC 로 정합성 보강
- 우리 analytics 서비스(Kafka Streams + ClickHouse) 는 이미 4단계에 진입한 케이스로, "초기 서비스가 성급하게 따라가지 않도록" 가이드를 명문화하는 것이 시니어 책임.

- 후원/협찬 멘트 없음 ("후원을 받아 제작" 류 표현 미발견)
- 외부 강의/제품/책/SaaS 홍보 없음
- 채널 자체 홍보(구독/좋아요/멤버십/강의 모집) 없음
- 도입(00:00) 인사 "안녕하세요. 코딩하는 기술사입니다." 와 마무리(12:55) "감사합니다." 외에는 전 구간이 본 주제(원본+보조 저장소 패턴)에 집중.

→ 본 요약은 **광고 제외 처리 없이 전체 콘텐츠를 그대로 분석**한 결과.
