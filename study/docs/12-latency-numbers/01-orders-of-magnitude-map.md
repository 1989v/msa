# 01. 자릿수 사다리 + 인접 비율

> Phase 1 의 유일한 deep file. **암기 ≠ 학습**. 사다리의 칸 사이 비율을 감각으로 가지면, 표에 없는 케이스도 추정할 수 있다.

## 0. 이 파일에서 얻을 것

- ns / µs / ms 그룹의 **8자리수 스펙트럼** 을 머릿속에 상주시킨다
- 인접 구간 **비율 5개** 를 외운다 (절대값은 외우지 않는다)
- "이 호출은 어느 자릿수?" 라는 질문에 0.5초 안에 답한다
- 이 감각이 왜 **아키텍처 의사결정의 베이스라인** 인지 이해한다

---

## 1. 왜 "절대값 암기" 는 권장되지 않는가

### 1-1. 숫자는 매년 변한다

Jeff Dean 의 LADIS 2009 키노트 표는 16년 전(2009년) 의 값이다. 2026년 현재:
- **SSD random read**: 2009 ~16µs → 2020 ~16µs (NVMe 보급 후 정체. 의외로 안 바뀜)
- **DRAM bandwidth**: DDR3 → DDR5 로 ~3배 향상
- **네트워크 1Gbps → 100Gbps**: NIC 자체 latency 는 비슷, throughput 은 100배

**Colin Scott 인터랙티브 페이지** 의 슬라이더가 이 변화를 시각화한다 — 2020 까지 갱신, 어떤 항목은 거의 안 변하고 어떤 항목은 자릿수가 바뀐다. **개별 숫자는 매년 outdated** 가 된다.

### 1-2. 그러나 자릿수 그룹은 잘 안 변한다

- L1 캐시는 2009년에도, 2026년에도 **ns 그룹**
- DRAM 은 항상 **~100ns** 근처 (자릿수가 바뀌지 않는다)
- HDD seek 은 물리적 한계로 **~ms 그룹** 에서 절대 못 벗어난다
- 광속은 1억 년 후에도 같다 → 대륙 간 RTT 는 영원히 **~100ms 그룹**

→ **그룹과 비율은 영구 자산**. 절대값은 휘발성.

### 1-3. 영상의 메시지 (사용자가 본 자료)

> "Don't memorize the numbers. Memorize the **ratios** between adjacent layers, and use them to **make architectural decisions**."

- 외워야 할 것: 칸과 칸 사이 비율 (예: L1 vs DRAM ≈ ×100)
- 활용할 것: 시스템 설계 토론에서 자릿수로 즉답 ("이건 µs, 저건 ms")
- 피해야 할 것: 표 전체를 Anki 카드로 만들어 외우기

## 2. Jeff Dean LADIS 2009 page 12 — 역사적 원전

원전 표(2009년 기준):

| 동작 | latency |
|---|---|
| L1 cache reference | 0.5 ns |
| Branch mispredict | 5 ns |
| L2 cache reference | 7 ns |
| Mutex lock/unlock | 25 ns |
| Main memory reference | 100 ns |
| Compress 1KB w/ Zippy | 3,000 ns (3 µs) |
| Send 2KB over 1Gbps NIC | 20,000 ns (20 µs) |
| Read 1MB sequentially from memory | 250,000 ns (250 µs) |
| Round trip within same datacenter | 500,000 ns (500 µs) |
| Disk seek (HDD) | 10,000,000 ns (10 ms) |
| Read 1MB sequentially from disk | 20,000,000 ns (20 ms) |
| Send packet CA → Netherlands → CA | 150,000,000 ns (150 ms) |

### 이 표의 천재성

1. **모든 단위를 ns 로 통일** → 비율 계산이 자명해진다 (250,000 / 100 = 2,500)
2. **자릿수 점프가 시각적** → "ns → µs → ms 로 1000배씩 올라감" 이 한눈에
3. **물리적 한계가 보임** → 광속/회전 디스크/NAND 의 차이가 숫자로 드러남

## 3. Colin Scott Interactive — 2020 기준 갱신

URL: https://colin-scott.github.io/personal_website/research/interactive_latency.html

### 슬라이더로 본 16년의 변화

- **거의 안 바뀐 것**: L1/L2/DRAM, mutex, HDD seek, 대륙 간 RTT
- **꽤 바뀐 것**: SSD (HDD 대비 자릿수 우위가 더 벌어짐), 같은 DC RTT (~500µs 유지), NIC throughput
- **자릿수 그룹 이동은 거의 없음**: ns 그룹은 ns, µs 는 µs, ms 는 ms 로 머무름

→ **결론**: 그룹핑은 향후 10-20년 더 유효할 가능성 높음.

## 4. 자릿수 사다리 (8자리 스펙트럼)

```
 1 ns  ▲ L1 cache ($1)                    ╲
        │  ×3                                │
 ~3 ns ─┤ L2 / 분기 예측 실패              │  ns 그룹
        │  ×30                               │ (CPU/메모리)
~100 ns ─┤ DRAM (random read)               │
        │  ×100                              │
        │                                    │
  ~1 µs ─┤ in-process 함수 호출 / 1KB 압축  ╱
        │  ×10                              ╲
 ~10 µs ─┤ 1KB over 1Gbps NIC                │
        │  ×2                                │  µs 그룹
 ~16 µs ─┤ SSD random read                   │ (디스크/근거리 net)
        │  ×30                               │
~500 µs ─┤ 같은 DC 내 RTT                   ╱
        │  ×4                               ╲
  ~2 ms ─┤ HDD seek                          │
        │  ×5                                │
 ~10 ms ─┤ 같은 리전 다른 AZ RTT            │  ms 그룹
        │  ×10                               │ (원거리 net)
~100 ms ─┤ 한국 → 미국 RTT                  │
        │  ×1.5                              │
~150 ms ▼ CA → Netherlands → CA             ╱
```

### 그룹 별 특징

| 그룹 | 범위 | 한계 요인 | 대표 케이스 |
|---|---|---|---|
| **ns** | 1 ns ~ 100 ns | CPU 클럭 / 전기 신호 거리 | 캐시, 메모리, mutex |
| **µs** | 1 µs ~ 500 µs | NAND 페이지 / NIC 처리 / DC 내부 거리 | SSD, 1Gbps NIC, 같은 DC RTT |
| **ms** | 1 ms ~ 200 ms | 회전 디스크 / 광속 한계 | HDD seek, 대륙 간 RTT |

> **트릭**: 외우기 어려우면 "ns 는 CPU 안", "µs 는 한 건물 안", "ms 는 도시/대륙 사이" 로 공간 비유로 묶는다.

## 5. 외울 비율 5개 (절대값 X)

```
1. L1 → DRAM         ≈ ×100      "캐시 친화적 코드는 100배 빠르다"
2. DRAM → SSD        ≈ ×1000     "메모리 캐싱은 디스크보다 1000배 압승"
3. SSD → DC 내 RTT   ≈ ×30       "Redis 가 로컬 SSD 보다 가끔 느린 이유"
4. DC → 대륙 간 RTT  ≈ ×300      "멀티 리전이 느린 이유 = 광속 한계"
5. 평균 → P99        ≈ ×3 ~ ×10  "tail latency 가 사용자 경험을 결정"
```

### 비율 5개를 고른 이유

- **#1 (L1→DRAM)**: CPU 캐시 친화 / cache miss 의 비용을 직관화
- **#2 (DRAM→SSD)**: "왜 캐시 레이어를 두는가" 의 핵심 근거
- **#3 (SSD→DC RTT)**: 분산 시스템에서 "원격 vs 로컬" 의 자릿수 차이. **Redis vs 로컬 디스크** 같은 헷갈리는 비교에 답 제공
- **#4 (DC→대륙 RTT)**: 멀티 리전 / 글로벌 사용자 / CDN 결정의 베이스라인
- **#5 (평균→P99)**: 단일 latency 가 아닌 분포의 **tail** 이 SLA 와 fan-out 곱셈을 지배

→ 5개로 시스템 설계의 80% 결정을 정량 추론할 수 있다.

## 6. 직관 트레이닝 — 5개 시나리오로 자가 검증

각 질문에 **자릿수만** 답하라. 절대값은 필요 없다.

### Q1. "Redis GET 한 번 vs MySQL PK lookup 한 번, 자릿수 차이?"
- Redis GET (네트워크 + 메모리 조회) ≈ DC 내 RTT 그룹 → **µs ~ 1ms**
- MySQL PK lookup (네트워크 + 인덱스 조회 + 디스크 가능성) → **ms 그룹**
- 차이 ≈ **×10 ~ ×100**
- → "Redis 캐시 hit 가 MySQL hit 보다 1-2 자릿수 빠르다"

### Q2. "한국 사용자가 미국 리전 API 를 호출하면, 응답까지 최소 얼마?"
- 단순 RTT 만 ≈ **~150 ms** (광속 한계)
- 거기에 서버 처리 ms + DB ms 추가 → 사용자 체감 **200~300 ms** 가 최소 시작점
- → "광속이 한계. CDN/엣지/리전 미러링이 답"

### Q3. "동기 RPC 호출 10개를 직렬로 부르면 P99 latency 는?"
- 단일 호출 P99 가 ~10 ms 면 → 직렬 합산 ≈ **~100 ms**
- 평균이 아닌 **P99 합산** 이라는 점이 함정 (평균 합 ×3~10 가 P99)
- → "병렬화 / 배칭 / 비동기로 자릿수 절약 가능"

### Q4. "100개 백엔드 fan-out 시 tail latency 는 평균보다 얼마나 늘어나는가?"
- 단일 호출 평균 5ms / P99 50ms 라면 → 100개 fan-out 시 **거의 항상 P99 영역** 진입
- 수학적으로 max(N개) ≈ P(99.99) → 단일 평균의 **×10 ~ ×30**
- → "fan-out 는 tail 의 곱셈. 백엔드 1개의 P99 가 전체를 결정"

### Q5. "ALB 헬스체크 1ms vs 같은 노드 in-process 호출 1µs, 같은 일을 하는데 왜 1000배?"
- ALB = 별도 프로세스 + 네트워크 스택 + TCP/TLS + L7 파싱 = **DC 내 RTT 그룹**
- in-process = 함수 호출 = **메모리 그룹**
- 차이는 "프로세스/네트워크 경계" 그 자체의 비용 — 자릿수 1000배
- → "마이크로서비스화 = 함수 호출이 네트워크 호출이 됨 = 자릿수 1000배 추가 비용"

## 8. Phase 2 로 넘어가기 전 자가 점검

다음 질문에 자릿수로 즉답할 수 있으면 Phase 2 준비 완료:

- [ ] "L1 캐시와 DRAM 의 비율?" → ×100
- [ ] "DRAM 과 SSD 의 비율?" → ×1000
- [ ] "같은 DC RTT 의 자릿수?" → µs 그룹 (~500 µs)
- [ ] "한국 → 미국 RTT?" → ms 그룹 (~150 ms)
- [ ] "P99 가 평균의 몇 배?" → ×3 ~ ×10
- [ ] "마이크로서비스화의 latency 비용?" → 함수 호출 → 네트워크 호출 = ×1000
- [ ] "왜 SSD 가 DRAM 보다 느린가?" → NAND 4KB 페이지 단위 vs 전기 신호 (Phase 2 에서 깊이 다룸)

자가 점검 미흡 항목은 표시해 두고 Phase 2 학습 중 보강.
