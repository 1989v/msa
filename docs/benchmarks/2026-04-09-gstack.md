## 1. Source Overview

gstack transforms Claude Code into a "virtual engineering team" through a rich collection of slash-command skills spanning the full software lifecycle: Think, Plan, Build, Review, Test, Ship, Reflect. Garry Tan reports 600,000+ lines written in 60 days using this system while running YC full-time.

Key distinguishing features:
- 30+ slash-command skills organized by lifecycle phase
- Real Chromium browser automation (daemon model, persistent sessions)
- Cross-model review (Claude + OpenAI Codex)
- Multi-agent orchestration (8 AI coding platforms supported)
- Persistent learning system (JSONL-based project memory)
- Template-driven documentation generation (SKILL.md from .tmpl files)

## 3. Patterns gstack Has That We Do Not

### 3.1 Persistent Learning System (HIGH VALUE)
gstack's `/learn` skill maintains a JSONL-based knowledge base per project:
- Typed entries (pattern, pitfall, preference, architecture, tool)
- Confidence scores (1-10)
- Append-only with deduplication by key
- Searchable, prunable, exportable
- Auto-surfaces top learnings at session start

**Current MSA state**: Static MEMORY.md file, manually maintained. No search, no confidence scoring, no automatic surfacing.

**Recommendation**: ADOPT. Build a `/learn` equivalent that stores project learnings in a structured format. Even a simplified version (JSONL file with type/key/insight/confidence) would improve cross-session knowledge retention significantly.

### 3.2 Multi-Pass Review with Cross-Model Validation (MEDIUM VALUE)
gstack's `/review` runs:
1. Scope drift detection
2. Critical pass (SQL safety, concurrency, auth)
3. Specialist subagent dispatch (testing, security, performance, migrations, API contracts)
4. Cross-model adversarial review (Claude + Codex)
5. Confidence-scored findings with deduplication

**Current MSA state**: Self-review protocol with L1/L2/L3 levels and fresh-context reviewer option.

**Recommendation**: PARTIAL ADOPT. The specialist dispatch pattern (security, performance, API contracts as independent sub-reviews) is worth adding to the self-review protocol. Cross-model review is interesting but adds complexity and cost.

### 3.3 Security Audit Skill (MEDIUM VALUE)
gstack's `/cso` combines OWASP Top 10 + STRIDE threat modeling as a dedicated review phase.

**Current MSA state**: No dedicated security review skill.

**Recommendation**: ADOPT. For a commerce platform handling orders and payments, a structured security review skill is directly valuable. Can be added as an agent-os standard or .claude/skills/ entry.

### 3.4 Shipping Pipeline Skills (MEDIUM VALUE)
gstack has `/ship` (test synthesis + coverage audit + PR automation), `/land-and-deploy` (merge + CI + health check), `/canary` (post-deploy monitoring), `/document-release` (auto-update docs).

**Current MSA state**: Manual build/deploy process.

**Recommendation**: SELECTIVE ADOPT. A `/ship` equivalent that runs build, tests, generates PR description, and checks coverage would save time. Canary and deploy are less relevant for local development.

### 3.5 Template-Generated Skill Docs (LOW-MEDIUM VALUE)
Skills use `.tmpl` template files that compile to SKILL.md via `bun run gen:skill-docs`. This prevents documentation drift.

**Current MSA state**: Hand-written SKILL.md files.

**Recommendation**: DEFER. Only 4 skills currently; template generation adds complexity without proportional benefit at this scale.

### 3.6 Design Workflow (LOW VALUE for this project)
`/design-shotgun` (iterative visual exploration), `/design-html` (production-ready HTML/CSS), `/design-consultation` (design system research).

**Recommendation**: SKIP. MSA is a backend-heavy microservices platform. code-dictionary has frontend but the design workflow is overkill.

### 3.7 Browser Automation (LOW VALUE for this project)
Full Chromium daemon with CDP, anti-bot stealth, cookie import, headless/headed modes.

**Recommendation**: SKIP. Backend microservices do not benefit from browser automation at this stage.

### 3.8 Retrospective Skill (LOW VALUE)
`/retro` with team-aware per-person metrics and shipping streaks.

**Recommendation**: SKIP. Solo developer context; retros are overkill.

## 4. Patterns We Have That gstack Does Not

### 4.1 Risk Classification with Automated Enforcement
MSA's L1/L2/L3 risk levels with Ralph Loop (max 3 retries, failure classification, escalation) is more systematic than gstack's ad-hoc /careful + /freeze + /guard approach.

**Assessment**: MSA's approach is more rigorous for high-stakes domains (commerce, payments). Keep.

### 4.2 ADR-Enforced Architecture Governance
The `adr-check.sh` hook automatically detects architecture-sensitive file modifications and injects relevant ADR context. gstack has no equivalent architectural governance mechanism.

**Assessment**: Critical for MSA's clean architecture constraints. Keep.

### 4.3 Compaction Management
Detailed compaction rules (when to compact, when not to, pre/post checklists), COMPACTION-GUIDE.md, and PrePrompt hooks for context usage monitoring (75%/90% warnings).

**Assessment**: gstack does not address context window management at all. MSA's approach is mature. Keep.

### 4.4 Per-Service Context Files
Each of 9 services has its own CLAUDE.md and docs/ directory, providing service-specific context loading.

**Assessment**: Essential for MSA's multi-service architecture. Keep.

### 4.5 Hierarchical Standards (agent-os/)
Structured standards directory with core-rules, confirmation, self-review, doc-gardening, compaction, session management, and global conventions.

**Assessment**: More organized than gstack's flat file approach. Keep.

### 4.6 PostToolUse Hooks for Quality Gates
Spec and tasks file write hooks that inject checklists (readability, independence, line limits).

**Assessment**: Proactive quality enforcement that gstack lacks. Keep.

## 6. Key Takeaways

1. **gstack is breadth-oriented; MSA harness is depth-oriented**. gstack covers the entire software lifecycle with 30+ skills but has shallow enforcement. MSA covers fewer phases but with deeper automation (hooks, risk levels, ADR checks).

2. **gstack's persistent learning is its strongest innovation**. The JSONL-based `/learn` system with confidence scoring, pruning, and auto-surfacing is a clear gap in the MSA harness. This is the highest-priority adoption candidate.

3. **MSA's governance model is superior for production systems**. Risk classification, ADR enforcement, and compaction management are critical for a commerce platform that gstack's "move fast" ethos does not prioritize.

4. **gstack's review system is more sophisticated but heavier**. Multi-pass, cross-model, specialist dispatch review is impressive but may be over-engineered for a solo/small team. The specialist dispatch pattern alone (independent security/performance/API sub-reviews) is worth extracting.

5. **Different target audiences explain different trade-offs**. gstack optimizes for maximum throughput by a single developer shipping across many projects. MSA's harness optimizes for correctness and architectural consistency in a complex domain.
