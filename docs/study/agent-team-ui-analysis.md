## TL;DR

- 같은 Claude Code를 써도 **하네스(Harness) 설계 수준**에 따라 생산성 격차가 극심해진다
- 하네스 엔지니어링은 프롬프트가 아닌 **에이전트가 일하는 환경 자체를 설계**하는 패러다임이다
- CLAUDE.md, Hooks, Skills/Plugins 등 하네스 구성 요소를 이미 실무에 적용하는 사람과 그렇지 않은 사람 사이의 격차가 벌어지고 있다

## 핵심 내용

### 1. 하네스 엔지니어링이란

하네스(Harness)는 원래 말의 힘을 안전하게 제어하기 위한 마구를 뜻한다. AI 에이전트 맥락에서는 **강력한 모델의 능력을 올바른 방향으로 제어하고 운용하기 위한 설계 구조**를 의미한다.

핵심 기능 세 가지:

| 기능 | 설명 |
|------|------|
| **제어(Control)** | 에이전트가 허용 범위 밖에서 동작하지 않도록 방지 |
| **모니터링(Monitoring)** | 에이전트의 행동과 출력을 실시간 추적/기록 |
| **피드백(Feedback)** | 오류를 감지하고 후속 동작에 개선 사항을 반영 |

### 2. "격차를 만드는" 핵심 요소

영상의 핵심 메시지는 **LLM Literacy**의 격차에 있다. 동일한 모델과 IDE를 사용해도 다음과 같은 차이가 발생한다:

**생산성이 높은 엔지니어 (하네스 활용자):**
- 리포지토리 가이드라인, 린트 규칙, 코드 패턴을 컨텍스트에 사전 로드
- CLAUDE.md에 아키텍처, 테스트 규칙, 패키지 규칙을 체계적으로 문서화
- Hooks로 에이전트 행동을 자동 제어 (예: main 브랜치 커밋 차단)
- Skills/Plugins로 반복 워크플로우를 슬래시 커맨드로 자동화
- 결과: 10분 만에 머지 가능한 코드 생산

**생산성이 낮은 엔지니어 (하네스 미활용자):**
- "이 함수 리팩토링해줘" 같은 컨텍스트 없는 요청
- 팀 컨벤션과 맞지 않는 결과물에 대한 수정 루프 반복
- 결과: 1시간 이상 소요, 추가 수정 필요

### 3. Claude Code 하네스의 구성 요소

Anthropic의 Claude Code가 제공하는 하네스 메커니즘:

#### 3.1. CLAUDE.md (프로젝트 지시서)

프로젝트 루트에 위치하는 구조화된 문서로, 에이전트에게 프로젝트의 규칙과 맥락을 제공한다.

포함해야 할 내용:
- 아키텍처 원칙 (Clean Architecture, 의존성 방향)
- 모듈 구조 및 빌드 규칙
- 패키지 네이밍 규칙
- 테스트 규칙 (프레임워크, 스타일, 위치)
- API 응답 포맷
- 인프라 설정 규칙

#### 3.2. Hooks (사전/사후 행동 제어)

에이전트의 특정 행동 전후에 자동으로 실행되는 스크립트:

- **사전 훅 예시**: git commit 시도 시 현재 브랜치가 main이면 "feature/ 브랜치를 생성하세요" 안내
- **사후 훅 예시**: 코드 생성 후 자동으로 린트/포맷 실행
- 권한 모델: 기본적으로 읽기 전용, 사용자가 명시적으로 승인해야 쓰기 가능

#### 3.3. Skills / Plugins (워크플로우 자동화)

반복적인 작업을 슬래시 커맨드로 캡슐화:

- `/new-feature`: 대화형 요구사항 수집 -> Jira 이슈 생성 -> 브랜치 생성 -> 구현 계획 수립
- `/analyze`: URL에서 콘텐츠 수집 -> 분석 -> 학습 문서 생성
- 1M 컨텍스트, 슈퍼파워, 에이전트 라우터 등 플러그인 생태계 확장 중

#### 3.4. 진행 상황 추적 (Progress Tracking)

장시간 에이전트 운용 시 세션 간 연속성을 확보하는 메커니즘:

- `claude-progress.txt`: 에이전트가 수행한 작업의 시간순 로그
- Feature list (JSON): 200개 이상의 세부 기능 명세와 완료 상태 추적
- Git commit history: 작업 기록의 자동 문서화

### 4. 멀티 에이전트 자동화 시스템

QJC 채널의 관련 풀버전 영상 "[AI 에이전트 11개가 알아서 일한다](https://www.youtube.com/@qjc_qjc)"에서 시연한 시스템 구조:

- 11개 이상의 에이전트가 역할별로 분담하여 협업
- 각 에이전트는 전문화된 하네스를 통해 제어
- Initializer Agent가 환경을 설정하고, Coding Agent들이 순차/병렬로 기능 구현
- 하네스가 에이전트 간 컨텍스트 공유와 충돌 방지를 담당

### 5. 조직 수준의 하네스 전략 (3계층 모델)

Toss 기술 블로그에서 제시한 조직 적용 모델:

| 계층 | 범위 | 내용 |
|------|------|------|
| **Global Layer** | 회사 전체 | 보안 정책, 기본 코딩 스타일 |
| **Domain Layer** | 팀/도메인 | 팀별 비즈니스 로직, 도메인 규칙 |
| **Local Layer** | 리포지토리 | 프로젝트별 구현 세부사항 |

이 3계층이 합쳐져 **Executable SSOT** (실행 가능한 단일 진실 원천)를 형성한다. 기존 문서화는 시간이 지나면 낡지만, 하네스는 플러그인으로 즉시 업데이트되고 팀 전원에게 동시 전파된다.

## 주요 인사이트

1. **패러다임 전환 - 프롬프트에서 환경으로**: "더 좋은 프롬프트"를 작성하는 것이 아니라 "더 좋은 환경"을 설계하는 것이 AI 에이전트 시대의 핵심 역량이다. 프롬프트 엔지니어링의 한계를 하네스 엔지니어링이 보완한다.

2. **이미 쓰고 있었다는 자각**: CLAUDE.md를 작성하고, hooks를 설정하고, skills를 활용하는 행위 자체가 이미 하네스 엔지니어링이다. 용어를 몰랐을 뿐, 실무에서 이미 실천하고 있는 개발자들이 있다. 이 인식의 차이가 격차를 만든다.

3. **RAG보다 명시적 코드**: RAG 시스템은 검색 점수가 불투명하고 가시성이 떨어진다. 반면 하네스(플러그인/CLAUDE.md)는 명시적 코드로 작성되어 개발자가 주입되는 컨텍스트를 100% 통제할 수 있다.

4. **데이터 플라이휠 효과**: 하네스를 통해 정규화된 워크플로우 데이터는 도메인 특화 LLM (sLLM)의 instruction-tuning 데이터셋이 된다. 하네스 성숙도가 높아질수록 더 나은 AI 모델을 만들고, 이는 다시 워크플로우를 개선하는 선순환 구조를 형성한다.

5. **조직 생산성의 저점 올리기**: 개인 간 생산성 격차를 줄이는 핵심은 개인 역량 향상이 아니라, 조직 지식을 실행 가능한 형태로 코드화하여 모든 구성원의 최소 생산성(저점)을 높이는 것이다.

## 실무 적용 포인트

- **CLAUDE.md 체계화**: 프로젝트의 아키텍처 원칙, 모듈 구조, 패키지 규칙, 테스트 규칙을 CLAUDE.md에 명시하라. 현재 이 프로젝트(`msa`)에는 이미 잘 구성된 CLAUDE.md가 존재하며, 이것 자체가 하네스 엔지니어링의 실천이다.
- **Hooks 활용**: settings.json에 사전/사후 훅을 설정하여 에이전트의 위험한 행동(main 직접 커밋, force push 등)을 자동으로 차단하라.
- **Skills/Plugins 도입**: 반복적인 워크플로우(기능 개발, 코드 리뷰, 디버깅 등)를 슬래시 커맨드로 자동화하라. 현재 프로젝트에 설치된 content-analyzer, ai-debugger, harness-scaffold 등이 그 예시이다.
- **진행 상황 추적 파일**: 장기 프로젝트에서 claude-progress.txt 또는 유사한 메커니즘을 도입하여 세션 간 컨텍스트 연속성을 확보하라.
- **팀 표준화**: 하네스를 개인이 아닌 팀/조직 수준으로 확장하라. Global/Domain/Local 3계층 구조로 지식을 계층화하면 팀 전체의 생산성 저점을 높일 수 있다.

## 비판적 분석

- **강점**: 하네스 엔지니어링이라는 개념을 짧은 Shorts 영상으로 압축하여 핵심 메시지("격차를 만드는 것은 모델이 아니라 하네스")를 효과적으로 전달한다. "나는 이미 쓰고 있었다"라는 자기 인식 프레이밍이 시청자의 행동 변화를 유도한다.
- **한계/주의점**: Shorts 포맷 특성상 구체적인 구현 방법이나 코드 예시는 포함되지 않는다. 하네스 엔지니어링의 학습 곡선과 초기 구축 비용에 대한 언급이 부족하다. "현 시점의 스냅샷에 불과"하다는 비판도 존재하며, 모델 발전에 따라 하네스 구조도 지속적 업데이트가 필요하다.
- **대안적 관점**: IDE 내장 AI(Cursor, Windsurf 등)의 접근법은 하네스를 IDE 수준에서 추상화하여 진입 장벽을 낮추는 방향이다. CLI 기반 하네스(Claude Code)와 GUI 기반 하네스(IDE 플러그인) 중 어느 쪽이 장기적으로 우세할지는 아직 미결 영역이다.
