Platform Overview

1. System Context

MSA-based commerce system.

Services: Gateway, Product, Order, Search, Charting, Common

Infrastructure:

  • MySQL (per-service, master/replica)
  • Redis (cluster-ready)
  • Kafka (event-driven)
  • Elasticsearch (Search-only)
  • PostgreSQL + pgvector (Charting-only, port 5433 -- ADR-002)

docs / architecture / 00.clean-architecture.md

Clean Architecture (Kotlin Version)

1. Purpose

  • Complete separation of business logic and technical details
  • Framework-replaceable structure
  • Maximum testability
  • Domain-centric design
  • Long-term maintainable codebase

2. Layer Structure

2.1 Dependency Direction (Absolute Rule)

Presentation
      ↓
Application
      ↓
    Domain
      ↑
Infrastructure
  • Dependencies always point inward (toward Domain)
  • Domain knows no external frameworks
  • Infrastructure handles technology at the outermost layer

3. Layer Responsibilities

3.1 Domain Layer

Responsibility: Core business rules, Aggregate Root, Entity, Value Object, Domain Policy, Domain Event, Business exceptions

Prohibited: Spring deps, JPA annotations, HTTP code, DB access, messaging, external API calls

3.2 Application Layer

Responsibility: Use case orchestration, transaction boundaries, Port interface calls, DTO-Domain conversion

Allowed: @Service, @Transactional

Prohibited: Technical implementation details, direct DB entity usage, business rule implementation

3.3 Infrastructure Layer

Responsibility: DB integration, external APIs, messaging, cache, file system, framework config

Role: Implements Port interfaces, converts technical exceptions to domain/application exceptions

3.4 Presentation Layer

Responsibility: HTTP request/response, input validation, auth, API docs

Prohibited: Business logic, DB access

4. Port & Adapter Pattern

4.1 Inbound Port (UseCase Interface)

Location: Application layer

interface CreateUseCase {
    fun execute(command: Command): Result
}

4.2 Outbound Port

Location: Application layer

interface RepositoryPort {
    fun save(aggregate: Aggregate): Aggregate
    fun findById(id: Long): Aggregate?
}

5. Package Structure (Nested Submodule)

{service}/
├── domain/          -- :{service}:domain (pure Kotlin)
│   └── src/main/kotlin/com/kgd/{service}/domain/
│       ├── model/ / policy/ / event/ / exception/
└── app/             -- :{service}:app (Spring Boot)
    └── src/main/kotlin/com/kgd/{service}/
        ├── application/ (usecase/ service/ port/ dto/)
        ├── infrastructure/ (persistence/ client/ messaging/ config/)
        └── presentation/ (controller/ dto/)

6. Data Flow Rules

Input: Request DTO → Command → UseCase → Domain → Port → Infrastructure

Output: Infrastructure → Domain → Result → Response DTO

7. Architecture Quality Checklist

  • Application → Infrastructure direct references: 0
  • Domain → Spring/JPA dependencies: 0
  • Circular references: 0
  • Domain test coverage: 90%+
  • Key use case integration tests: 100%

  • Direct queries require API calls
docs / architecture / communication.md

Communication Strategy

1. Synchronous Communication

Technology: WebClient

Targets: Gateway → internal services, external API calls

Rules: CircuitBreaker required, Timeout mandatory, Retry policy explicit

2. Asynchronous Communication

Technology: Kafka

Purpose: Domain event propagation, search index updates, reduced coupling

3. Kafka Topic Naming

Format: {domain}.{entity}.{event}

Examples: product.item.created, order.order.completed

4. WebClient Standard Pattern

  • Base URL separation
  • Timeout settings
  • CircuitBreaker applied
  • Common error mapping

5. CircuitBreaker Standard

  • Resilience4j
  • Failure-rate based open
  • Half-open state support
  • Minimal fallback
docs / architecture / data-strategy.md

Data Strategy

1. Database Isolation Principle

  • Per-service DB isolation
  • Schema sharing prohibited
  • Direct DB access prohibited

2. MySQL Read/Write Separation

Structure: Master (Write) + Replica (Read)

Write transactions go to Master. Reads go to Replica. Strong consistency reads use Master.

3. Transaction Boundary

  • Transactions valid only within a service
  • Distributed transactions prohibited
  • Event-based eventual consistency

4. Query Strategy

  • JPA-based
  • QueryDSL for complex queries
  • Read model separation considered for complex reads
docs / architecture / resilience-strategy.md

Resilience Strategy

1. Circuit Breaker Policy

Resilience4j, sliding window failure rate, fast failure on Open state

2. Timeout Policy

  • External API: short timeout
  • Internal services: defined SLA
  • DB: connection pool management

3. Retry Policy

  • Retry only for idempotent requests
  • Exponential backoff
  • No infinite retries

4. Failure Propagation Prevention

  • Kafka async decoupling
  • Bulkhead pattern considered
  • Cache fallback strategy

5. High Availability Strategy

  • Stateless services
  • Horizontal scaling
  • Redis Cluster design
  • Elasticsearch replicas
docs / architecture / cdc-pipeline.md

CDC Pipeline Extension Guide

Overview

CDC (Change Data Capture) reads DB binary logs in real-time and streams change events to Kafka. It can replace or complement the search-batch DB direct read approach (ADR-0009).

Comparison with Current Architecture

AspectKafka App Events (Current)DB Direct Read (ADR-0009)CDC (Extension)
Data completenessEvent loss if code missedReads DB directlyCaptures all DB changes
Real-timemsBatch schedulems~seconds
Service boundaryCompliantException allowedCompliant (via Kafka)
Ops complexityLowLowHigh (Kafka Connect)

Recommended Adoption Timing

  • When product count exceeds tens of millions and batch indexing becomes a bottleneck
  • When real-time search reflection becomes an SLA requirement
  • When Kafka Connect infrastructure is already in operation

Architecture

MySQL (product_db) --binlog--> Debezium MySQL Connector
                                      | (Kafka Connect)
                                      v
                          Kafka Topic: dbz.commerce.products
                                      |
                          +-----------+-----------+
                          v                       v
                 search-consumer           search-batch
              (real-time incremental)    (initial/full index)

Implementation Priority

  1. Current (done): App events + API/DB batch
  2. Short-term: MySQL binlog + Debezium local test
  3. Mid-term: CDC consumer addition (parallel)
  4. Long-term: Full CDC transition
docs / adr / ADR-0001-multi-module-gradle.md

ADR-0001: Multi-Module Gradle Project Structure Accepted

Context

6 independent services (common, discovery, gateway, product, order, search) managed in a single repository. Shared library (common) needed. Independent build/deploy required.

Decision

  • Gradle multi-module + Version Catalog (gradle/libs.versions.toml)
  • Root build.gradle.kts declares common plugins/dependencies only
  • Each service module independently generates bootJar
  • common module generates jar only (no bootJar)

Consequences

  • Shared Gradle build cache improves speed
  • common module changes require all services to rebuild
  • Centralized dependency version management via libs.versions.toml
docs / adr / ADR-0002-language-and-framework.md

ADR-0002: Language & Framework Selection Accepted

Decision

  • Java 25 LTS (JVM runtime), Kotlin 2.2.21 (primary language)
  • Spring Boot 4.0.3 (Spring Framework 7.x)
  • Spring Cloud 2025.1.0
  • WebFlux adoption prohibited (except Gateway)
  • Coroutines limited to external IO (WebClient)
  • JPA + QueryDSL (blocking DB access)
  • Kotest BehaviorSpec (BDD testing)
  • JWT HS256 -- Gateway verifies JWT exclusively

Consequences

  • Kotlin all-open plugin required
  • kotlin-jpa plugin required for JPA entities
  • Spring Boot 4.0 = Jakarta EE 11 (no javax.*)
docs / adr / ADR-0003-inter-service-communication.md

ADR-0003: Inter-Service Communication Accepted

Decision

  • Sync: WebClient + Resilience4j CircuitBreaker
  • Async: Kafka event-based
  • Topic naming: {domain}.{entity}.{event}
  • CircuitBreaker: sliding window, 50% failure rate → Open
  • Timeout: external 3s, internal 5s
  • Retry: max 3, exponential backoff, idempotent only

Consequences

  • Strong consistency impossible → Eventual Consistency
  • Kafka Consumer Group horizontal scaling
  • DLQ design tracked separately
docs / adr / ADR-0004-authentication-and-authorization.md

ADR-0004: Authentication & Authorization Accepted

Decision

  • JWT (HS256): Access Token 30min, Refresh Token 7d (Redis)
  • JWT Blacklist in Redis (logout/force expiry)
  • Gateway verifies JWT, passes X-User-Id/X-User-Roles headers
  • JWT secret via environment variable (never hardcoded)

Consequences

  • Redis Cluster failure → Fail-Open policy with monitoring/alerting
  • Pre-expiry token invalidation only via blacklist
  • Internal services accessible only through Gateway

ADR-0005: Service Discovery Accepted

docs / adr / ADR-0006-database-strategy.md

ADR-0006: Database Strategy Accepted

Decision

  • Per-service independent MySQL instances (product-db, order-db fully separated)
  • Master (Write) + Replica (Read) per service
  • AbstractRoutingDataSource for @Transactional(readOnly=true) → Replica routing
  • JPA + QueryDSL, blocking I/O
  • No distributed transactions → Kafka eventual consistency
  • Schema migration: Flyway

ADR-0007: Cache Strategy (Redis) Accepted

Decision

    • Session: session:refresh:{userId} (TTL 7d)

ADR-0008: Search Strategy (Elasticsearch) Accepted

docs / adr / ADR-0009-search-batch-db-direct-access.md

ADR-0009: Search Batch DB Direct Access Exception Accepted

Context

search-batch full index uses Product API. API offset pagination causes page drift during changes. Snapshot consistency matters more for large-scale initial indexing and DR recovery.

Decision

Allow search-batch module read-only JDBC access to mysql-product-replica.

  • Read-only (SELECT only) to replica only
  • Activated only when reindex.source=db
  • Default: reindex.source=api
  • Two separate Spring Batch Jobs: productApiReindexJob / productDbReindexJob

Consequences

  • search-batch couples to product DB schema
  • Separate DB credentials required
  • Supersedes ADR-0008 "no MySQL" rule for search-batch only
docs / adr / ADR-001-charting-service-introduction.md

ADR-001: Charting Service MSA Introduction Accepted

Context

Stock OHLCV chart pattern similarity analysis is a completely separate domain from the commerce platform. Requires custom embedding (32-dim) and pgvector cosine similarity search.

Decision

Add charting as a new MSA service: Port 8010 (API), 5433 (PostgreSQL+pgvector), 3010 (Frontend). Independent Docker Compose deployment. Clean Architecture applied.

Consequences

  • Positive: Domain isolation, independent deploy/scale, technology freedom
  • Negative: Additional service operations, separate PostgreSQL management
docs / adr / ADR-002-pgvector-over-elasticsearch.md

ADR-002: pgvector over Elasticsearch Accepted

Decision

Use pgvector (PostgreSQL extension) for vector storage: vector(32) column, HNSW index (m=16, ef_construction=64, vector_cosine_ops).

Single PostgreSQL instance manages metadata + vectors + OHLCV data together.

Alternatives Rejected

  • Elasticsearch: Search-service-only infra, can't be shared (DB isolation rule)
  • Pinecone/Weaviate: SaaS cost, vendor lock-in
  • FAISS: No persistence, sync required
docs / adr / ADR-003-python-fastapi-language-choice.md

ADR-003: Python/FastAPI Language Choice Accepted

Decision

Python 3.11 + FastAPI for charting service (data science requirements).

  • ORM: SQLAlchemy 2.x (async)
  • Numerics: numpy, pandas, scipy
  • Data: yfinance, FinanceDataReader
  • Scheduler: APScheduler 3.x
  • Migration: Alembic

Rationale

yfinance, FinanceDataReader are Python-ecosystem-only. JVM lacks equivalent numerical computing libraries. Polyglot microservices is a standard MSA pattern.

Package Structure Convention

Base Package

com.kgd.{service}

Nested Submodule Layout

{service}/
├── domain/                    -- :{service}:domain (pure Kotlin)
│   └── src/main/kotlin/com/kgd/{service}/domain/
│       └── {entity}/
│           ├── model/         # Aggregate, Entity, VO
│           ├── policy/        # Domain Policy, Specification
│           ├── event/         # Domain Event
│           └── exception/     # Domain Exception
└── app/                       -- :{service}:app (Spring Boot)
    └── src/main/kotlin/com/kgd/{service}/
        ├── application/{entity}/
        │   ├── usecase/       # UseCase interface (Inbound Port)
        │   ├── service/       # UseCase impl
        │   ├── port/          # Outbound Port interface
        │   └── dto/           # Command, Result, Query
        ├── infrastructure/
        │   ├── persistence/{entity}/
        │   │   ├── entity/    # JPA Entity
        │   │   ├── repository/# Spring Data + QueryDSL
        │   │   └── adapter/   # RepositoryPort impl
        │   ├── client/        # WebClient adapters
        │   ├── messaging/     # Kafka adapters
        │   └── config/        # Tech config
        └── presentation/{entity}/
            ├── controller/    # RestController
            └── dto/           # Request/Response DTO

Rules

  1. domain module: No Spring/JPA annotations (compile error)
  2. Cross-reference prohibited: Order → Product direct reference forbidden, API call only
  3. app module: implementation(project(":{service}:domain"))
  4. bootJar name: tasks.bootJar { archiveBaseName.set("{service}") }

docs / plans / 2026-03-09-search-pipeline.md

Search Pipeline Implementation Plan

Goal: Add search:consumer (incremental indexing) and search:batch (full indexing) modules with BulkIngester-based ES pipeline.

Architecture:

  • search:app -- read-only REST search API (write responsibility removed)
  • search:consumer -- Kafka event → BulkIngester async incremental indexing
  • search:batch -- Spring Batch + WebClient(product:app) → BulkIngester → alias swap full indexing
  • Both modules use Dual BulkIngester pattern (primary 5s/1000 + retry 3s/500)

Pattern Chart UI Implementation Plan

Goal: Implement lightweight-charts based professional candlestick chart with 10 pattern overlays + technical indicator toggles + future trend projection.

Architecture: All pattern library, matching, indicator calculations are client-side (no additional API). Lightweight Charts with direct DOM manipulation. Main chart (candle+overlay) and sub-charts (Volume/RSI/MACD) as separate components with subscribeVisibleLogicalRangeChange time axis sync.

Tech Stack: lightweight-charts ^4.2, React 18, TypeScript, date-fns

Tasks Overview

  1. Install lightweight-charts
  2. Create patterns.ts (10 pattern definitions)
  3. Create patternMatcher.ts (Pearson correlation)
  4. Create indicators.ts (MA, BB, RSI, MACD)
  5. Create IndicatorToggle.tsx
  6. Create PatternSelector.tsx
  7. Create PatternInfoBar.tsx
  8. Create PatternChart.tsx (main chart)
  9. Rewrite App.tsx
  10. Remove legacy components
  11. End-to-end verification
