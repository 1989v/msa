## 1. 개요

박성현의 『세븐 스플릿』 분할 투자 전략을 빗썸/업비트 등 국내 암호화폐 거래소에 적용하는 자동매매 웹 서비스. 자본을 7개 회차(확장 가능)로 분할하고, 각 회차가 **독립 매수가를 기준으로 독립 익절**하는 방식으로 변동성을 수익 기회로 전환한다.

세븐스플릿 본가에는 주식 대상 "매직스플릿"과 암호화폐 대상 "비트스플릿" 제품 라인이 존재하며, 본 서비스는 공개된 전략 원리(박성현 7원칙)를 참고해 **개인이 자기 자본으로 직접 운용하는 형태의 웹 어플리케이션**을 목표로 한다.

### 박성현 7원칙 (공개 자료 기준)

1. 장기투자 계좌의 투자 자산 비중은 전체의 **40% 이상** 유지
2. **레버리지(신용/미수) 사용 금지**
3. 장기투자 계좌 목표 수익률 **10% 이상**
4. 개별 종목 최초 매수 금액은 해당 계좌 자산의 **5% 이내**
5. 추가 매수는 **직전 회차 매수가 대비 -3% 이상 손실 시**에만
6. 추가 매수 투자금 규모는 **최초 매수 금액과 동일**
7. **손절매 없음**

매도 원칙: **각 회차 매수가 대비 수익 시 해당 회차만 독립 매도** (전체 평균단가 기준 아님).

## 2. 핵심 기능

### 2.1 전략 엔진 (세븐스플릿 정통 규칙)

박성현 7원칙을 알고리즘화한 분할매매 엔진. 주식 기준 수치를 암호화폐 변동성에 맞춰 configurable하게 외부화.

**사용자 입력 (라운드 단위 설정)**
- 대상 종목 (예: BTC/KRW)
- 라운드 총 투입 금액
- 스플릿 회차 수 (기본 7, 1~50 범위)
- 회차별 매수 금액 (기본값 = 총 투입금액 / 회차수, 원칙 6에 따라 모든 회차 동일)
- 매수 간격 % (기본 -3%, 원칙 5)
- 매도 기준 % (회차별 독립 익절 %, 기본값 세트는 회차별로 다르게 지정 가능)

**동작 알고리즘**
1. 전략 시작 → 현재 시세로 **1회차 매수** (회차별 매수 금액만큼)
2. **직전 회차 매수가 대비 -n% 하락 시 다음 회차 매수** (원칙 5·6)
3. 전체 회차 소진 시까지 (2) 반복
4. **각 회차는 해당 회차 매수가 대비 +m% 도달 시 독립 매도** (평균단가 기준 아님)
5. 매도된 회차는 슬롯 비움 → 이후 추가 하락 시 해당 회차 슬롯에 재매수 가능
6. **손절 없음** (원칙 7) — 전 회차 소진 후 추가 하락 시 대기

**파라미터 외부화 (`SplitStrategyConfig`)** — 회차 수, 매수 간격 %, 회차별 익절 % 전부 configurable. 주식 기준(-3%/+10%)을 암호화폐에 맞게 튜닝(예: -5~-8% / +5~+15%)하는 것은 백테스트로 결정.

### 2.2 시세 수신 & 실행 인프라

- 거래소 **WebSocket** 구독을 1차 경로, REST 폴링(매초)을 폴백으로 사용
- 수신된 틱을 메모리 이벤트 버스로 전파 → 해당 종목을 구독 중인 모든 전략 인스턴스에 전달
- 유저 수 × 유저당 종목 수만큼 동시 모니터링 필요 → **가상 스레드(Java 21 Loom) 또는 Kotlin 코루틴** 기반 non-blocking 런타임 필수

### 2.3 기타 기능

- **거래소 연동**: 빗썸 / 업비트 OpenAPI (잔고 조회, 주문, 체결 조회)
- **대상 종목 선정**: 사용자가 거래쌍 직접 선택 (자동 추천은 차순위)
- **Phase별 모드**: 백테스트 → 페이퍼 트레이딩 → 실매매
- **대시보드**: 회차별 포지션, 실현/미실현 손익, 누적 수익률, 매매 이력 시각화
- **리스크 관리**: 전체 포트폴리오 손실 한도, 거래쌍별 최대 투입 금액, 긴급 청산 스위치
- **Rate Limiter**: 빗썸·업비트 API 호출 한도는 **유저 API Key 단위**로 적용되므로 유저별 격리만 잘 하면 시스템 전체로는 여유. 유저 단위 호출량 카운터로 한도 근접 시 백오프
- **알림**: 체결/손익/이상 상황 알림 (푸시 또는 메일)

## 3. 사용자 시나리오

1. **전략 셋업**: 사용자가 거래소 API 키 등록 → 대상 거래쌍 선택 → 총 투입 금액과 분할 유닛 수, 매수/매도 %포인트 설정 → 전략 활성화
2. **자동 운용**: 엔진이 실시간 시세를 감시하면서 조건 충족 시 유닛 단위로 매수/매도 주문 자동 실행, 사용자는 대시보드로 진행 상황만 확인
3. **리포트 & 조정**: 일/주 단위로 누적 수익률·유닛별 체결 내역 리포트 확인, 필요 시 파라미터(유닛 수, 매수 간격 등) 재조정 후 재가동
4. **긴급 중단**: 시장 급락 등으로 위험 감지 시 사용자가 "전략 중지" 또는 "전량 청산" 버튼을 눌러 즉시 대응

## 4. 기술 스택

- **[결정]** **백엔드**: Kotlin 2.2.21 + Spring Boot 4.0.3 (ADR-0002 준수), Java 25 LTS 런타임
- **[결정]** **런타임 모델**: ADR-0002 혼용 패턴
  - 컨트롤러·도메인·내부 DB(JPA/QueryDSL)는 **blocking** 유지
  - **거래소 REST 호출은 WebClient + Kotlin Coroutine `suspend`**
  - **거래소 WebSocket 시세 구독은 Coroutine `Flow`**
  - Tomcat 요청 처리는 **가상 스레드 활성화** (`spring.threads.virtual.enabled=true`) — 전략 인스턴스 대량 수행 시 스레드 병목 방지
  - WebFlux 도입 금지 (ADR-0002)
- **실시간 시세 수집**: 거래소 WebSocket 구독 1차 / REST 매초 폴링 폴백 → 내부 이벤트 버스(Kafka 또는 in-process)로 전파
- **[결정]** **전략 엔진 상태 관리**: 하이브리드 패턴 — 현재 상태는 CRUD 테이블(`round_slot`)로 읽기 최적화, 모든 상태 변경은 도메인 이벤트(`OrderPlaced`, `OrderFilled`, `SlotOpened`, `SlotClosed` 등)로 outbox 패턴 append-only 기록 후 Kafka 발행. 사후 재현·감사·analytics 연동·백테스트 재계산 가능
- **저장소**: MySQL (주문/체결/회차 상태), ClickHouse (시세 틱·백테스트 시계열) — 기존 MSA 인프라 재활용
- **프론트엔드**: React 기반 대시보드 (차트는 `charting` 서비스와 협업 가능)
- **보안**: 거래소 API Key 암호화 저장 (AES-GCM 봉투 암호화), IP 화이트리스트
- **인프라**: 기존 MSA 공통 플랫폼 (K8s / MySQL / Kafka / ClickHouse) 위에 신규 서비스 추가

## 5. 서비스 구조 (예상)

```
seven-split/
├── domain/                     # 유닛, 전략, 주문 도메인 모델
│   ├── strategy/               # 세븐스플릿 전략 규칙
│   ├── unit/                   # 유닛(분할 슬롯) 집계
│   └── risk/                   # 손익 한도/중단 규칙
├── app/
│   ├── api/                    # REST: 전략 CRUD, 상태 조회, 긴급 중단
│   ├── stream/                 # WebSocket 시세 구독 + Kafka 발행
│   ├── executor/               # 주문 집행기 (거래소 REST 호출)
│   ├── backtest/               # 과거 데이터 기반 시뮬레이션
│   └── scheduler/              # 일/주 리포트 생성
└── adapters/
    ├── bithumb/                # 빗썸 API 어댑터
    └── upbit/                  # 업비트 API 어댑터
```

기존 MSA 플랫폼과의 관계:
- `gateway`: 인증/라우팅
- `member` / `auth`: 사용자 식별·권한
- `analytics`: 매매 이벤트 수집/분석 (선택)
- `charting`: 가격·지표 차트 제공 협업 (선택)

## 6. 미결 사항 / 질문

### 전략 규칙
- **[결정 2026-04-24]** 박성현 7원칙 정통 구현 (리서치 결과 반영)
  - 매수 트리거: **직전 회차 매수가 대비 -3% 하락 시** (원칙 5, configurable)
  - 매수 금액: **모든 회차 동일 금액** (원칙 6)
  - 매도: **각 회차 매수가 대비 독립 익절** (Q5 → A 확정)
  - 손절 없음 (원칙 7)
  - 트리거는 고정 % (ATR 동적은 향후 옵션)
  - 파라미터 전부 configurable
- **[결정]** 1회차 최초 진입 = 전략 시작 즉시 시장가 매수 (옵션으로 "고점 대비 -x% 대기" 트리거 추후 추가 가능)
- **[결정]** 전 회차 소진 후 추가 하락 시 **대기** (원칙 7 손절 없음) — 회차 중 하나라도 익절되면 슬롯이 비고, 다시 매수 트리거 동작
- **[미결]** "2분의 1 포인트" 용어의 출처 — 공개 자료엔 명시 없음. 사용자가 책/영상에서 본 구체 정의가 있으면 공유 필요. 없다면 위 규칙으로 갈음.
- **[미결]** 암호화폐 변동성 기준 기본값 제안 — 주식 기준(-3%/+10%)을 직접 쓸지, 암호화폐 조정값(예: -5~-8%/+5~+15%)으로 시작할지. 백테스트로 결정.

### 거래소 연동
- **[결정 2026-04-24]** 빗썸·업비트 **공식 OpenAPI(REST + WebSocket)** 연동. 인증 방식이 상이(빗썸: API Key + Secret HMAC-SHA512, 업비트: JWT)하므로 `ExchangeAdapter` port 인터페이스로 격리.
- **[결정 2026-04-24]** MVP 우선순위 = **빗썸 먼저 (Phase 1~2)**, **Phase 3 실매매 진입 시 업비트 추가**. 어댑터 port는 1일차부터 추상화해 두 거래소 교체/병렬 실행이 가능하도록 설계.
- **[미결]** 주문 실패/부분체결/네트워크 장애 시 재시도 및 상태 복구 규칙 — 기본 방침은 idempotent order ID + exponential backoff. 구체 정책은 구현 단계에서 확정.
- **[부분 결정]** Rate Limit 관리 = 유저 API Key 단위로 호출량 카운터, 한도 근접 시 백오프. 빗썸·업비트 각자 공식 한도값은 구현 시점에 확인.
- 해외 거래소(Binance 등) 확장은 국내 운용 실적 확인 후 검토 (장기 로드맵).

### 1번(퀀트 트레이더)과의 관계
- **[결정 2026-04-24]** **완전 독립 서비스**로 확정. 1번(LLM 멀티 에이전트 토론 기반)과 12번(결정론 규칙 기반)은 매매 패러다임이 정반대라 공유할 의사결정 로직이 없음. 거래소·시세 수신 같은 실행 인프라는 미래에 1번이 실제로 구현될 때 공통 모듈(`trading-core`)로 리팩터링할 수 있으나, 현 시점엔 오버엔지니어링이라 보류. 12번의 거래소 어댑터는 처음부터 clean port로 설계해두어 나중에 뽑기 쉽게 유지.

### 법/보안
- 국내에서 개인이 자기 자본으로 운용하는 자동매매 도구의 법적 이슈 검토
- API Key 저장/접근 통제, 감사 로그 요건
- 타인에게 전략을 제공/대행하는 형태로 확장할 경우 금융규제 검토 필요

### 제품 범위
- **[결정 2026-04-24]** 하이브리드 전략 채택 — 코드는 multi-tenant SaaS 구조로 설계하되 운영 초기에는 본인 계정 1개만 사용. `member`/`auth` 연동은 1일차부터 포함. 외부 오픈 시점은 개인 운용 실적 확인 후 판단.
- **[결정 2026-04-24]** 단계적 출시 채택 — **Phase 1 백테스트(과거 틱 기반 시뮬레이션) → Phase 2 페이퍼 트레이딩(실시간 시세 + 가상 체결) → Phase 3 실매매**. 과거 틱 데이터는 기존 MSA의 ClickHouse 재활용. 각 Phase는 독립 릴리즈 가능하도록 모듈화.
