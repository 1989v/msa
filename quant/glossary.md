## 1. Bounded Context Overview

Quant BC는 **자동매매 전략(분할·시그널·융합) + 차트 분석 + 입문자 지표 학습 CMS** 통합 플랫폼 (ADR-0033). 자산 클래스 무관 (Asset / Market 추상화) — 암호화폐와 주식 동일 인터페이스. Phase 1 = 빗썸 암호화폐 + sealed Strategy + paper trading. Phase 2 = 실거래(Live). Phase 3 = Hybrid + Audit chain + KillSwitch + RiskLimit (ADR-0037).

⚠️ **commerce 도메인의 Order 와 본 BC의 Order/OrderCommand 는 의미 완전히 다름** — 본 BC는 거래소 주문.

## 2. Aggregates & Entities

### Strategy (sealed interface)
**Type.** Aggregate (sealed interface, marker)
**Definition.** 자동매매 전략의 sealed marker (ADR-0033). 모든 전략 타입의 공통 식별자 노출.
**Subtypes.** TrancheStrategy / SignalStrategy / HybridStrategy
**Invariants.** id (StrategyId), tenantId (TenantId) 공통 필드만 강제. 자산/거래소는 자식이 보유.
**Code.** `com.kgd.quant.domain.strategy.Strategy`

### TrancheStrategy
**Type.** Aggregate
**Definition.** 7분할 진입 전략. **상태 전이는 반드시 Aggregate 메서드 경유** (ADR-0022).
**Lifecycle.** `INITIAL` → `ACTIVE ↔ PAUSED` → `LIQUIDATED`
**Methods (각각 DomainEvent 반환).**
- `activate()` → `StrategyActivated`
- `pause()` → `StrategyPaused`
- `resume()` → `StrategyResumed`
- `liquidate(reason)` → `StrategyLiquidated`
**Invariants.** `status.ensureTransition(target)` — 위반 시 `IllegalStrategyTransitionException`
**Code.** `com.kgd.quant.domain.strategy.TrancheStrategy`

### SignalStrategy
**Type.** Aggregate (data class)
**Definition.** Single-source 시그널 기반 전략 (ADR-0033 Phase 1). VolumeSpike / RsiBreakout / MaCross / BollingerSqueeze 등.
**Invariants.**
- `market.supports(asset.assetClass)` — 거래소가 자산 클래스를 지원해야 함
- `exitSignal == null` 이면 진입만 수행, 청산은 외부(수동/RiskLimit) 위임
**Code.** `com.kgd.quant.domain.strategy.SignalStrategy`

### HybridStrategy
**Type.** Aggregate (data class)
**Definition.** TrancheStrategy + SignalStrategy 합성 (ADR-0033 Phase 3). signalGate trigger 시 trancheBase 분할 진입 시작.
**Invariants.**
- `trancheBase.tenantId == tenantId` (3개 식별자 일치)
- `signalGate.tenantId == tenantId`
- `signalGate.asset == asset`
- `signalGate.market == market`
**Code.** `com.kgd.quant.domain.strategy.HybridStrategy`

## 3. Value Objects

### Asset
**Type.** Value Object (data class)
**Definition.** 시계열 가격이 정의되는 거래 대상. 자산 클래스 무관 추상화 (ADR-0033).
**Fields.** `code: AssetCode`, `assetClass: AssetClass` (CRYPTO/STOCK_KR/STOCK_US), `displayName`
**Invariants.** `displayName.isNotBlank()`
**Code.** `com.kgd.quant.domain.asset.Asset`

### Market
**Type.** Value Object
**Definition.** 시세/주문이 발생하는 거래소 또는 데이터 소스. BITHUMB / UPBIT / BINANCE / YAHOO / FDR_KR.
**Invariants.**
- `supportedClasses.isNotEmpty()` — 최소 1개 자산 클래스 지원
- `displayName.isNotBlank()`
**Method.** `supports(assetClass)` — 자산 클래스 지원 여부
**Code.** `com.kgd.quant.domain.market.Market`

### OrderCommand (🔴 commerce Order와 다름)
**Type.** Value Object (data class)
**Definition.** **거래소로 보내는 주문 실행 요청**. Application → Exchange Adapter 전달.
**Fields.** orderId / side / type / quantity / price?(Market 주문은 null) / strategyId? / symbol?
**Avoid.** Order (commerce BC와 혼동 절대 금지)
**Code.** `com.kgd.quant.domain.order.OrderCommand`

### Execution
**Type.** Value Object (data class)
**Definition.** 거래소의 체결 이벤트 한 건. 부분 체결 다건 가능.
**Fields.** price + quantity + executedAt
**Code.** `com.kgd.quant.domain.order.Execution`

### RiskLimit
**Type.** Value Object (data class with init)
**Definition.** 테넌트별 위험 한도 (ADR-0037 Phase 3). 일일 손실/거래량/단일주문 최대 KRW.
**Invariants.**
- 3개 한도 모두 > 0
- `singleOrderMaxKrw <= dailyVolumeLimitKrw` — 단건이 일일 한도 초과 금지
**Defaults.** 100k / 1M / 100k (보수적)
**Methods.** `lossBreached(currentLoss)`, `volumeBreached(currentVolume)`, `singleOrderExceeds(orderKrw)`
**Reset.** 매일 KST 00:00
**Code.** `com.kgd.quant.domain.live.RiskLimit`

### AuditEvent
**Type.** Value Object (data class with SHA-256 chain)
**Definition.** Phase 3 audit chain 이벤트 (ADR-0037). 변조 방지 hash chain.
**Algorithm.** `currentHash = SHA-256(prevHash || canonical(payload) || occurred_at)`. 매일 KST 03:00 verify job.
**Invariants.**
- `currentHash.length == 64` (SHA-256 hex)
- `prevHash == null || prevHash.length == 64`
- `payloadCanonical.isNotBlank()`
- canonical JSON (정렬된 키)
**Code.** `com.kgd.quant.domain.live.AuditEvent`

### 기타 식별자/값 객체
- `StrategyId`, `OrderId`, `RunId`, `TenantId`, `AssetCode`, `MarketCode`, `SlotId` — 도메인 ID VO
- `Price`, `Quantity`, `Percent` — 도메인 수치 VO
- `Symbol`, `Tick`, `PriceTick`, `Orderbook` — 시세 표현

## 6. Policies & Invariants

### KillSwitch (sealed interface) — 3 레벨
**Type.** Policy (비상 정지)
**Levels (Subtypes).**
| Level | Scope | Storage |
|---|---|---|
| `Global` | 모든 테넌트 | Redis: `quant:kill-switch:global` |
| `Tenant` | 특정 tenantId | Redis: `quant:kill-switch:tenant:{tenantId}` |
| `Strategy` | 특정 strategyId | Redis: `quant:kill-switch:strategy:{strategyId}` |

**Rule.** 어느 레벨이라도 enabled=true → 해당 범위의 신규 주문 차단.
**Latency.** Redis ≤200ms 반영.
**Audit.** MySQL `kill_switch_log` (append-only).
**Code.** `com.kgd.quant.domain.live.KillSwitch`

### 도메인 예외 → invariant 추출

| Exception | 의미 |
|---|---|
| `IllegalStrategyTransitionException` | Strategy 상태 전이 위반 |
| `IllegalSlotTransitionException` | TrancheSlot 상태 전이 위반 |
| `LeverageAttemptException` | 레버리지 시도 차단 (Phase 1 = spot only) |
| `StopLossAttemptException` | 도메인 정책 위반 손절 시도 |
| `TrancheStrategyConfigInvalidException` | TrancheStrategyConfig 유효성 위반 |
| `BacktestRunNotFoundException` | id 조회 실패 |
| `StrategyNotFoundException` | id 조회 실패 |
| `NotImplementedInPhase1Exception` | Phase 1 범위 외 호출 |

## 11. Cross-Context Integration (🔴 매우 중요)

### Shared Terms with conflict

| 용어 | 본 BC | 다른 BC | 해결 |
|---|---|---|---|
| **Order / OrderCommand** | 거래소 주문 명령 | order BC: 고객 주문 (commerce) | **🔴 의미 완전 다름** — 본 BC는 OrderCommand 명시, 외부와 통신 시 절대 단순 "Order" 사용 금지 |
| **Execution** | 거래소 체결 | order BC의 OrderStatus.COMPLETED 와 의미 다름 | 본 BC만 사용 |
| **OrderStatus** | 거래소 주문 상태 | order BC도 OrderStatus 보유 (PENDING/COMPLETED/CANCELLED) | 다른 enum, 패키지로 구분 |
| **Asset / Market** | 본 BC 전용 추상화 | commerce 영역엔 없음 | — |
| **Strategy** | sealed (Tranche/Signal/Hybrid) | — | 본 BC 전용 |
| **Audit** | hash chain audit | code-dictionary 의 audit 로그와 의미 다름 | 본 BC는 변조 방지 chain |

### External integrations
- Bithumb / Binance (거래소 REST + WebSocket)
- Yahoo Finance (시세 + 펀더멘털)
- Naver News (뉴스)

- Telegram Bot (알림)
- OCI Vault (KMS)
- ClickHouse (시세·감사 적재)
- Redis (저지연 state)

## 12. Flagged Ambiguities

| # | 항목 | grilling 질문 |
|---|---|---|
| 1 | OrderCommand vs commerce Order 명칭 | "외부 문서/API 응답에서 단순 'Order' 노출 시 commerce 와 혼동 위험 — 항상 'Quant Order' 또는 'OrderCommand' 강제?" |
| 2 | Strategy sealed 확장 | "Phase 3 이후 추가 전략 타입 추가 시 sealed 갱신 + 영속화 마이그레이션 정책?" |
| 3 | KillSwitch trigger 후 재시작 | "Off 토글 시 자동 재개 vs 수동 승인? 어떤 actor 권한?" |
| 4 | RiskLimit override 2FA | "기본값 override 시 2FA 필수 (KDoc) — 구현 위치 명시 필요?" |
| 5 | AuditEvent verify 실패 대응 | "매일 03:00 verify job 이 chain break 감지 시 — 알림? 자동 격리? 운영자 대응?" |
| 6 | KMS DEK rotation 정책 | "KekVersionLabel + LazyReencryptionJob — rotation 주기/조건?" |
| 7 | Phase 1 의 LeverageAttempt/StopLoss 차단 의도 | "현재는 spot only — 향후 활성화 정책?" |
| 8 | TenantId 다중 테넌트 격리 수준 | "헤더 기반 — 향후 DB 분리 가능성? 데이터 leak 방지 정책?" |
| 9 | OrderCommand strategyId/symbol nullable | "Phase 1 백테스트만 null 허용 — Phase 2 어댑터 require 가드 강제?" |
| 10 | Symbol vs AssetCode 차이 | "Symbol 은 거래소 표현(BTC/KRW), AssetCode 는 도메인 식별(BTC) — 매핑 책임?" |
